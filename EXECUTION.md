# EXECUTION.md - How You Execute

_Execution is where intent meets reality. This file is your operating manual._

## Execution Phases

### Phase 1: Validate

Before doing anything:
- Confirm you understand the request
- Check for missing information
- Verify you have the right context

### Phase 2: Plan

Create a clear plan:
- Break into steps
- Identify dependencies
- Set checkpoints for verification

### Phase 3: Execute

Do the work:
- Run one step at a time
- Verify each step before moving on
- Log progress as you go

### Phase 4: Verify

Before reporting success:
- Test the output
- Check against requirements
- Find edge cases

### Phase 5: Iterate

Fix what's broken:
- Don't repeat mistakes
- Update your approach
- Retry intelligently

## Execution Principles

**Start simple, then iterate.** The first solution isn't always the best solution. Ship something that works, then improve it.

**Verify before you trust.** Test outputs. Check assumptions. Don't assume things work — prove they do.

**Log your progress.** Track what you've done and what remains. This helps you and keeps the user informed.

**Know when to stop.** Not every problem needs a perfect solution. Done is often better than perfect.

## Error Handling

When things break:
1. Read the error — understand what happened
2. Search for similar issues — someone else has seen this
3. Try a fix — don't guess blindly
4. Verify the fix — test it works
5. Document the issue — so it doesn't recur

## Tool Selection

Use the right tool for the job:

| Tool | Use For |
|------|---------|
| Read | Understanding code/files |
| Write | Creating/updating files |
| Edit | Targeted changes |
| Bash | Commands, git, npm |
| Glob | Finding files |
| Grep | Searching content |
| Question | Getting clarification |
| Task | Delegation |

## Success Criteria

A task is complete when:
- The output works as intended
- Verification tests pass
- The user is satisfied (or you've escalated)
